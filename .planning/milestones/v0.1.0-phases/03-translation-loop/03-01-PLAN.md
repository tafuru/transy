<objective>
Build the translation foundation behind the popup: minimal text normalization, availability preflight,
short inline error mapping, and a request-scoped coordinator that makes stale completions impossible.

Purpose: Phase 3 fails if PopupView talks directly to Apple Translation without a guardrail layer.
This plan creates the contracts and tests that let the later popup wiring stay race-safe, inline-only,
and locked to generic English until Phase 4.

Output: `Transy/Translation/` foundation files plus three Swift Testing files covering preflight,
state transitions, and stale-result suppression.
</objective>

<task type="auto" tdd="true">
  <name>Task 3: Implement request-scoped translation coordination and stale-write guards</name>
  <files>Transy/Translation/TranslationCoordinator.swift</files>
  <behavior>
    - `begin(sourceText:)` creates a fresh request identity and publishes `.loading`
    - `finish(...)` publishes `.result` only when the request ID still matches the active request
    - `fail(...)` publishes `.error` only when the request ID still matches the active request
    - A second `begin(...)` supersedes the earlier request immediately
    - `dismiss()` clears active request identity so later completions are ignored
  </behavior>
  <action>
Implement `TranslationCoordinator.swift` as a `@MainActor` `@Observable` state machine for popup content.

Requirements:
- Define a nested `PopupState` enum that carries `requestID`, `sourceText`, and either translated text or error message
- Keep the coordinator framework-agnostic: no stored `TranslationSession`, no macOS 26-only convenience APIs, no detached long-lived translation task
- Expose only the request/state transitions PopupView needs later (`begin`, `finish`, `fail`, `dismiss`)
- Treat request identity as the fake translation seam required by validation: tests should be able to simulate success/failure by calling `finish`/`fail` directly without Apple language assets

Do NOT collapse stale handling into “cancel and hope.” Every result/error write must explicitly guard against
an outdated `activeRequestID`, because cancellation alone is not reliable enough for the locked stale-result behavior.
